# Mermaid sidecar convention

This repo (and any repo that adopts this pattern) keeps mermaid diagrams as **sidecar `.mmd` source files** alongside the `.md` items, with a committed `.svg` generated by `@mermaid-js/mermaid-cli`. The `.md` references the `.svg` via a plain image link. **Never** put raw ```` ```mermaid ```` blocks in the committed `.md` — they don't render in Ketryx-generated documents.

## Detect the convention

A repo uses this convention if any of these are true:
- It has a `Makefile` whose targets call `mmdc`.
- It has `@mermaid-js/mermaid-cli` in `package.json`.
- It has `.mmd` files committed next to `.md` files.
- It has `.githooks/pre-commit` that re-renders `.mmd` files.

If none of these are present, this skill doesn't apply.

## Rules when authoring or editing items

### Adding a diagram

1. Create `<item-name>.mmd` next to the `.md` item with the raw mermaid source. Use a descriptive suffix if the item has multiple diagrams: `spec-foo-flow.mmd`, `spec-foo-state.mmd`.
2. Run `make` (or `./node_modules/.bin/mmdc -i <item-name>.mmd -o <item-name>.svg --outputFormat svg --quiet`) to generate `<item-name>.svg`.
3. In the `.md` item, add an image link: `![Caption describing the diagram](<item-name>.svg)`.
4. Optionally add a `Source:` line so reviewers know where the source lives: `` Source: [`<item-name>.mmd`](<item-name>.mmd). ``
5. Stage all three files (`<item-name>.mmd`, `<item-name>.svg`, the `.md`) in the same commit.

### Editing an existing diagram

1. Edit the `.mmd` file — never the `.svg` directly.
2. Re-render via `make` (or `mmdc -i <name>.mmd -o <name>.svg ...`).
3. Stage both the `.mmd` and updated `.svg`.

### Anti-patterns to avoid

- **Do not** add ```` ```mermaid ```` blocks to committed `.md` files. They will not render in Ketryx document exports.
- **Do not** edit the `.svg` by hand. It will be overwritten on the next `make` and `make check` will flag it.
- **Do not** add a single inline `<details>` source block — the convention is sidecar `.mmd` files, not inline source preservation.
- **Do not** rename a `.mmd` without also renaming the matching `.svg` and updating every `.md` link to it.

## Verification

Before declaring an item-editing task done, verify:
1. Every `.mmd` you edited has a fresh `.svg` next to it.
2. `make check` exits 0 (re-renders every `.mmd` and diffs against the committed `.svg`).
3. Every `.md` image link resolves to a file that exists.

## Why this convention exists

Ketryx parses `.md` files as configuration items and includes them in generated documents. Raw ```` ```mermaid ```` blocks don't render at document-export time — they appear as code. Pre-rendering to `.svg` solves that.

The sidecar pattern (instead of a custom script that rewrites `.md` files) is chosen because:
- The only validated artifact is `mmdc` (the official upstream mermaid CLI).
- No custom code to validate, audit, or version-control as a regulated artifact.
- Each file (`.mmd`, `.svg`, `.md`) has its own discrete change history — easier for change control.
- A reviewer can read the `.mmd` to see what changed without running anything.

The trade-off — two files per diagram instead of one inline block — is small, and arguably *better* for traceability in a regulated environment.
