# Git-based item mermaid diagrams

How to use [mermaid](https://mermaid.js.org/) diagrams inside Ketryx Git-based configuration items.

## The problem

Ketryx parses Markdown files in your Git repo as configuration items and includes them in generated documents. Raw ```` ```mermaid ```` code blocks **don't render** in those documents — they appear as code.

## The pattern

For each diagram, commit two files side-by-side:

- `<name>.mmd` — the mermaid source.
- `<name>.svg` — the rendered image, generated by [`@mermaid-js/mermaid-cli`](https://github.com/mermaid-js/mermaid-cli).

The `.md` item references the `.svg` via a plain image link. Run `make` to (re)generate every `.svg` from its `.mmd`.

```
items/
├── spec-sensor-state-machine.md       ← references spec-sensor-state-machine.svg
├── spec-sensor-state-machine.mmd      ← mermaid source
└── spec-sensor-state-machine.svg      ← generated by `make`
```

The repo has no rendering code of its own. The `Makefile` and the pre-commit hook are thin wrappers around `mmdc`.

## Quickstart

```bash
git clone https://github.com/Ketryx/git-based-item-mermaid-diagrams.git
cd git-based-item-mermaid-diagrams
npm install                       # installs mermaid-cli at the pinned version
make                              # renders every items/*.mmd into items/*.svg
git config core.hooksPath .githooks   # enable the pre-commit re-render hook (one-time)
```

Open `items/spec-sensor-state-machine.md` — the image link resolves to the SVG you just generated.

## Authoring a diagram

1. Create `items/<name>.mmd` with your mermaid source.
2. Commit it — the pre-commit hook re-renders the matching `.svg`. (Or run `make` manually.)
3. In the `.md` item, reference the SVG: `![Caption](<name>.svg)`.
4. Commit the `.md`, `.mmd`, and `.svg` together.

**Naming.** One `.mmd` per diagram, named after the item it illustrates.

**Multiple diagrams in one item.** Give each diagram its own `.mmd` and a distinct suffix. For example, `items/spec-build-pipeline.md` has two diagrams:

```
items/
├── spec-build-pipeline.md             ← references both SVGs
├── spec-build-pipeline.mmd            ← stage flowchart source
├── spec-build-pipeline.svg
├── spec-build-pipeline-trigger.mmd    ← trigger sequence source
└── spec-build-pipeline-trigger.svg
```

The `.md` references each one with a plain image link. Non-mermaid images (screenshots, photos, hand-authored diagrams) live alongside without a `.mmd` — the convention only manages the mermaid ones.

**Editing.** Always edit the `.mmd`, never the `.svg`. The pre-commit hook re-renders the `.svg` from the `.mmd` whenever you stage one.

## Adopting in your own repo

Copy these files:

- `Makefile`
- `package.json`
- `.githooks/pre-commit`
- `.github/workflows/check-mermaid.yml`

If your items live somewhere other than `items/`, adjust the `find items` path in the `Makefile`. Then run `npm install` and `git config core.hooksPath .githooks`.

## Why it works for validated software

In a regulated environment, every script in the build chain is something you have to audit and change-control. This pattern minimizes that surface:

- The only third-party tool is `mmdc`, an upstream release.
- The `Makefile` is six lines and the pre-commit hook is twenty.
- Each artifact (`.mmd`, `.svg`, `.md`) is a discrete file with its own change history.
- The pre-commit hook re-renders the `.svg` whenever the `.mmd` changes, so source and image stay in sync at commit time.
- `make check` runs in CI to verify every `.mmd` has a committed `.svg` and still renders without errors.

## SVG vs. PNG

SVG diffs cleanly under git, scales without artifacts, and renders correctly in the Ketryx UI. If you export documents to DOCX and need diagrams rendered there, switch `--outputFormat svg` to `png` in the `Makefile` — Word's SVG support is inconsistent.

## Determinism

`@mermaid-js/mermaid-cli` is pinned in `package.json`, so every contributor renders with the same version. Mermaid's text layout depends on the host's installed fonts, so two machines rendering the same `.mmd` can produce SVGs with slightly different coordinates — committing the rendered `.svg` (rather than re-rendering at view time) keeps the document deterministic for reviewers.
