# Git-based item mermaid diagrams

How to use [mermaid](https://mermaid.js.org/) diagrams inside Ketryx Git-based configuration items.

## The problem

Ketryx parses Markdown files in your Git repo as configuration items and includes them in generated documents. Raw ```` ```mermaid ```` code blocks **don't render** in those documents — they appear as code.

## The pattern

For each diagram, commit two files side-by-side:

- `<name>.mmd` — the mermaid source.
- `<name>.svg` — the rendered image, generated by [`@mermaid-js/mermaid-cli`](https://github.com/mermaid-js/mermaid-cli).

The `.md` item references the `.svg` via a plain image link. Run `make` to (re)generate every `.svg` from its `.mmd`.

```
items/
├── spec-sensor-state-machine.md       ← references spec-sensor-state-machine.svg
├── spec-sensor-state-machine.mmd      ← mermaid source
└── spec-sensor-state-machine.svg      ← generated by `make`
```

That's the whole pattern. No custom rendering code in the repo — the `Makefile` is six lines, the pre-commit hook is twenty. The only tool involved is the official `mmdc` binary.

## Quickstart

```bash
git clone https://github.com/Ketryx/git-based-item-mermaid-diagrams.git
cd git-based-item-mermaid-diagrams
npm install                       # installs mermaid-cli at the pinned version
make                              # renders every items/*.mmd into items/*.svg
git config core.hooksPath .githooks   # enable the pre-commit re-render hook (one-time)
```

Open `items/spec-sensor-state-machine.md` — the image link resolves to the SVG you just generated.

## Authoring a diagram

1. Create `items/<name>.mmd` with your mermaid source.
2. Commit it — the pre-commit hook re-renders the matching `.svg`. (Or run `make` manually.)
3. In the `.md` item, reference the SVG: `![Caption](<name>.svg)`.
4. Commit the `.md`, `.mmd`, and `.svg` together.

**Naming.** One `.mmd` per diagram, named after the item it illustrates. If an item has multiple diagrams, suffix each one (`spec-foo-flow.mmd`, `spec-foo-state.mmd`).

**Editing.** Always edit the `.mmd`. Never edit the `.svg` by hand — `make check` will flag it as out of sync.

## Adopting in your own repo

Copy these files:

- `Makefile`
- `package.json`
- `.githooks/pre-commit`
- `.github/workflows/check-mermaid.yml`

Adjust the `find items` path in the `Makefile` if your items live elsewhere, then `npm install` and `git config core.hooksPath .githooks`. You're set.

## Why it works for validated software

In a regulated environment, every script in the build chain is something you have to audit and change-control. This pattern keeps that surface near zero:

- The only validated tool is `mmdc` (an upstream release).
- The `Makefile` and pre-commit hook are short enough to read in one screen.
- Every artifact (`.mmd`, `.svg`, `.md`) is a discrete file with its own change history.
- The pre-commit hook regenerates the `.svg` whenever the `.mmd` changes, so source and image stay in sync at every commit.
- `make check` runs in CI as a safety net — every `.mmd` must have a committed `.svg` next to it and must still render cleanly.

## SVG vs. PNG

SVG diffs cleanly under git, scales without artifacts, and renders correctly in the Ketryx UI. If you export documents to DOCX and need diagrams rendered there, switch `--outputFormat svg` to `png` in the `Makefile` — Word's SVG support is inconsistent.

## Determinism

`@mermaid-js/mermaid-cli` is pinned in `package.json`. Bumping it may change SVG output slightly; CI will surface the diff so you can review and re-commit.
