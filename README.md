# Git-based item mermaid diagrams

A worked example for using [mermaid](https://mermaid.js.org/) diagrams inside Ketryx Git-based configuration items, designed for **validated software development** environments.

## The problem

Ketryx parses Markdown files in your Git repo as configuration items, and includes their content in generated documents. However, raw ```` ```mermaid ```` fenced code blocks aren't rendered at document-export time — they appear as code, not as diagrams.

## The pattern (validated-friendly)

Every diagram is **two files** committed side-by-side:

- `<name>.mmd` — the mermaid source (the artifact under change control).
- `<name>.svg` — the rendered image (generated from the `.mmd`).

The `.md` configuration item references the `.svg` via a plain Markdown image link. To generate or refresh the `.svg`, you run `make`. That's it.

The only tool involved is the **official** [`@mermaid-js/mermaid-cli`](https://github.com/mermaid-js/mermaid-cli), pinned to a known version in `package.json`. There is no custom rendering code in this repo — the `Makefile` is a 6-line wrapper that calls `mmdc` once per `.mmd` file.

This matters in regulated environments: you don't have to validate a custom script. The validation surface is `mmdc` itself (an upstream tool with its own release process) and a Makefile small enough to read in one screen.

## What's in this repo

| Path | Purpose |
|---|---|
| `items/*.md` | Sample Ketryx items (Requirement, Software Item Specs) with image links to diagrams. |
| `items/*.mmd` | Mermaid source for each diagram. **This is the artifact under change control.** |
| `items/*.svg` | Generated by `make`. Committed so Ketryx can render the item docs without a build step. |
| `Makefile` | Six lines that call `mmdc` once per `.mmd`, plus a `check` target for CI. |
| `package.json` | Pins `@mermaid-js/mermaid-cli` so every contributor and CI runs the same version. |
| `.githooks/pre-commit` | Re-renders any staged `.mmd` so its `.svg` stays in sync. Activated by `git config core.hooksPath .githooks`. |
| `.github/workflows/check-mermaid.yml` | CI: runs `make check` to catch any `.svg` that doesn't match its `.mmd`. |

## Try it locally

```bash
git clone https://github.com/Ketryx/git-based-item-mermaid-diagrams.git
cd git-based-item-mermaid-diagrams
npm install        # installs mermaid-cli at the pinned version
make               # renders every items/*.mmd into items/*.svg
```

Then open `items/spec-sensor-state-machine.md` — the image link resolves to the generated SVG.

To enable the pre-commit hook (so edited `.mmd` files re-render automatically on commit):

```bash
git config core.hooksPath .githooks
```

One-time, per clone. The hook is a 20-line shell script — no `husky` or other tooling.

## Authoring a new diagram

1. Create `items/<name>.mmd` with the mermaid source.
2. Run `make` (or just commit — the pre-commit hook will render).
3. Reference the SVG from your `.md` item: `![Caption](<name>.svg)`.
4. Commit `<name>.mmd`, `<name>.svg`, and the `.md` item together.

The convention: **one `.mmd` per diagram**, named after the item it illustrates (`spec-sensor-state-machine.mmd` for `spec-sensor-state-machine.md`). When an item has multiple diagrams, suffix each one: `spec-foo-flow.mmd`, `spec-foo-state.mmd`.

A Claude skill at [`.claude/skills/mermaid-sidecar/`](.claude/skills/mermaid-sidecar/) teaches Claude this convention so it does the right thing when authoring or editing items.

## Adopting in your own repo

1. Copy `Makefile`, `package.json`, `.githooks/pre-commit`, and `.github/workflows/check-mermaid.yml` into your repo.
2. Adjust the `find items` path in the `Makefile` if your items live elsewhere.
3. `npm install` and `git config core.hooksPath .githooks`.
4. Author diagrams as `.mmd` sidecar files alongside your `.md` items.

## How `make check` works

For every `.mmd` file:
1. Render it to a temp SVG.
2. Diff against the committed `.svg`.
3. If they differ — or if the `.svg` is missing — fail the build.

This guarantees the committed `.svg` always reflects the current `.mmd` source. There's no way for source and image to drift out of sync without CI catching it.

## Determinism

- `@mermaid-js/mermaid-cli` is **pinned** in `package.json`. Bumping the version may change SVG output slightly; CI will flag the diff so you can re-render and review.
- All sample diagrams in this repo are byte-stable across re-runs (verified across `make clean && make` cycles).

## Why SVG and not PNG

SVG diffs cleanly under git, scales without artifacts, and renders correctly in the Ketryx UI at any zoom level. The trade-off: Microsoft Word's SVG support is inconsistent. If you export Ketryx documents to DOCX and need diagrams rendered, change `--outputFormat svg` to `png` in the `Makefile`.
